Sprint 1

Contrato de API e limpeza do legado

Fechar o risco mais alto primeiro: alinhar a API publicada com a superficie real e remover referencias operacionais que ainda tratam superficies em sunset como se fossem ativas.

Saida esperada: lista completa de modulos, base paths e handlers publicados.
Saida esperada: matriz router x endpoint x status de documentacao.
Criterio: o documento nao pode continuar restrito aos 3 paths atuais.
Evitar doc parcial ou enganosa para integradores e QA.
Registrar o progresso no arquivo docs/technical-debt/tracker.json.
Ponto de partida: 132 referencias encontradas em docs, README e package.json.
O runbook precisa apontar para o fluxo canonico atual de banco.
Os SLOs devem estar alinhados com docs/service-catalog.md.
Especial atencao para dev:legacy, stack:hybrid e textos de onboarding.
Sem isso, o tracker continua sub-representando a divida real do codigo.
Sprint 2

Reducao de custo de manutencao

Atacar a duplicacao de front-end e os hotspots de responsabilidades misturadas que tornam mudancas simples em alteracoes caras, acopladas e propensas a regressao.

Meta: reduzir a duplicacao hoje medida em 87.5% nesse componente.
Essas telas hoje repetem estrutura, layout e parte da logica de acao.
Meta: reduzir os clones internos e simplificar manutencao do detalhe de agentes.
Hoje o componente mistura carga remota, agregacao, retry, renderizacao e debug visual.
Objetivo: reduzir acoplamento entre interacao de UI e chamada remota.
O arquivo concentra validacao, parsing, callback GET/POST, auth context e sync em um unico modulo.
Separar claramente callback, sync, listagem e upsert melhora teste e leitura.
Refatoracao sem protecao de teste so desloca a divida tecnica.
Sprint 3

Type-safety e quality gates

Depois de reduzir o custo visivel de manutencao, fechar a fragilidade dos contratos internos e endurecer os gates quantitativos do CI.

O arquivo atual contem 17 ocorrencias de any em entidades centrais do dominio.
Esses tipos precisam parar de vazar como caixa-preta entre pacotes.
Criterio: contrato unico reaproveitavel entre pacotes do monorepo.
Hoje so existe piso explicito para Python com --cov-fail-under=80.
O gate deve falhar quando cobertura cair abaixo do piso aprovado.
Evitar rollout amplo sem baseline inicial.
Estrutural

Sinal de qualidade e baseline

Melhorar a confiabilidade das metricas para que a priorizacao futura use sinal util, e nao ruido vindo de arquivos gerados ou superficies ausentes do HEAD.

Hotspots de codigo gerado distorcem o ranking de refatoracao.
O core ativo nao deve competir com legado ausente no mesmo ranking operacional.
Produzir um baseline que reflita o codigo realmente mantido pelo time.
Isso fecha o ciclo entre relatorio, tracker e visibilidade executiva.

Criterios de saida

  • OpenAPI cobre a superficie canonica relevante de apps/api.
  • Runbooks e SLOs nao dependem mais de superficies em sunset como referencia operacional primaria.
  • O tracker passa a refletir os eixos reais de divida observados no codigo.
  • Os principais hotspots de UI deixam de depender de copia manual para evolucao.
  • packages/shared-types deixa de propagar any nas entidades centrais.
  • O CI passa a rejeitar quedas de cobertura em TypeScript nas areas criticas.
  • Os relatorios de qualidade passam a ignorar arquivos gerados e superficies fora do escopo ativo.

Observacoes de uso

  • Marque cada item apenas com evidencia objetiva em PR, artefato ou relatorio regenerado.
  • Se uma tarefa for grande demais para um unico PR, quebre em subtarefas tecnicas mantendo este checklist como backlog mestre.
  • O item so deve ser considerado concluido quando o tracker e a documentacao derivada tambem forem atualizados.
  • Itens P1 devem ter owner ativo no sprint atual; itens P2 precisam data alvo visivel; itens P3 podem entrar como lote de saneamento.
