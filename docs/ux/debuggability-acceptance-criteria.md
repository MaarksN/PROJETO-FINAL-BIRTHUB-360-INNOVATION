# Acceptance Criteria: Workflow Debuggability UI

To ensure the BirthHub360 platform is maintainable by tenant administrators, the Workflow Debugging UI must meet the following acceptance criteria before being marked "ready for production."

## 1. Visual Execution Path
*   **[ ] Criterion**: The UI must display a visual representation (DAG/Flowchart) of the entire workflow.
*   **[ ] Criterion**: Each step in the graph must have a clear visual indicator of its final state:
    *   Green = Success
    *   Red = Failed
    *   Grey = Skipped / Not Executed
    *   Yellow/Spinning = Currently Running or Paused
*   **[ ] Criterion**: The UI must clearly show which branch of a `ConditionStep` was taken.

## 2. Minimum Required Information (Per Step)
When a user clicks on any specific step in the execution graph, a detail panel must open containing the following *minimum* information:

*   **[ ] Execution Timestamps**: Exact `started_at` and `completed_at`/`failed_at` timestamps (in the user's local timezone), plus the total duration in milliseconds/seconds.
*   **[ ] Resolved Inputs**: The exact variable values passed *into* the step, after any string interpolation or data transformation was applied.
*   **[ ] Raw Outputs / Responses**: The exact data generated by the step. For HTTP actions, this must include the HTTP Status Code, Response Headers, and the raw Response Body.
*   **[ ] Error Stack Trace (If Applicable)**: If the step failed due to a system error (not just a 400 HTTP response), the engine's error message must be displayed (sanitized to remove internal code paths, focusing on the logical error).

## 3. Contextual Search and Navigation
*   **[ ] Criterion**: A user must be able to search the entire execution payload (JSON) for a specific string (e.g., an email address) to see which step introduced or modified it.
*   **[ ] Criterion**: The UI must provide a direct "Link to this Step" URL, allowing an admin to share the exact point of failure with a colleague via chat.

## 4. Actionability
*   **[ ] Criterion**: For failed runs (not in a terminal expired state), the UI must present a clear "Retry from Here" button.
*   **[ ] Criterion**: If a step requires payload modification before retrying (e.g., fixing a typo in an email address), the UI must provide a JSON editor to edit the input state before clicking Retry.

## 5. Security and Redaction
*   **[ ] Criterion**: Verification that variables marked as "sensitive" in the template definition appear as `********` in the debug UI inputs and outputs, confirming the redaction logic from `docs/security/debug-sensitive-data.md` is functioning.
