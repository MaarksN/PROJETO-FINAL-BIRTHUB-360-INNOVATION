Categoria auditada: Dívida de Dependência

Repositório: Birth Hub Innovation 360

Escopo auditado: Manifestos, overrides e lockfile que definem a versão efetiva de dependências do front-end canônico.

Quantidade de itens comprovados: 1

Status final da análise: Achados comprovados

Data/hora da análise: 2026-04-14T16:11:14.638Z

Método

A varredura comparou a versão declarada no pacote web, o override raiz e a resolução efetiva no lockfile. O item abaixo é uma divergência objetiva entre os artefatos de dependência versionados.

Resultado

Versão de Next diverge entre manifesto, override e lockfile

Diagnóstico: O pacote web declara uma versão. O override raiz impõe outra faixa. O lockfile resolve uma terceira referência efetiva. O artefato instalado não corresponde ao que o manifesto do app diz revisar.

Evidências:
  • apps/web/package.json:20 declara `"next": "16.1.6"`.
  • package.json:214 define override raiz `"next": ">=16.1.7"`.
  • pnpm-lock.yaml:267-269 registra para `apps/web` o specifier `>=16.1.7` com resolução `16.2.3(...)`.
  • pnpm-lock.yaml:10 e 2882 também carregam `next: '>=16.1.7'` como faixa efetiva.

O Juro (Impacto): Debug de regressão, revisão de upgrade e reprodução de build ficam ambíguos. Quem lê o manifesto do app não vê a mesma versão que realmente chega ao lockfile e ao runtime.

Plano de Quitação:
  1. Escolher uma única política de versão para `next` no workspace.
  2. Alinhar manifesto do app, override raiz e lockfile para a mesma decisão.
  3. Usar o gate de lockfile para impedir nova deriva após regenerar a resolução.
Prompt de Execução/Correção 
Papel
Você é um engenheiro de plataforma responsável por alinhar a governança de dependências do front-end canônico.

Missão
Eliminar a deriva de versão de `next` entre manifesto, override e lockfile.

Escopo
Atue somente sobre os arquivos que definem ou resolvem a dependência `next` no workspace.

Arquivos-alvo
- apps/web/package.json
- package.json
- pnpm-lock.yaml

Correções obrigatórias
- Definir primeiro a versão alvo única para `next` no core web.
- Aplicar a mesma decisão no manifesto, override e lockfile.
- Documentar a razão da versão escolhida se houver governança específica.

Restrições
- Não deixar ranges diferentes para o mesmo pacote no mesmo workspace.
- Não alterar dependências não relacionadas.
- Não regenerar lockfile com mudança colateral desnecessária.

Critérios de aceite
- Manifesto, override e lockfile apontam para a mesma política de versão.
- A instalação reproduz o que o pacote do app declara.
- O gate de governança de lockfile continua passando.

Validação
- pnpm install --frozen-lockfile
- pnpm ci:lockfile
- pnpm build
